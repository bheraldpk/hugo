Getting Started

CommureOS can be a bit overwhelming when you first try to install and run locally, so the goal of this is to simplify the process of onboarding and increase the overall efficiency of developers.

Architecture

Understanding the architecture at a high level will be helpful context to have before trying to install the core-platform. There are a few resources that will be helpful:

  • Architecture Diagram
  • Session 1 of our knowledge sharing series goes into the high level architecture. The first hour is important context that will set you up for success in the subsequent sessions.

At it’s core, CommureOS is a federated architecture. The apollo router is the core of the architecture. This is what enables different data sources to be served as a singular GraphQL API. From consumers perspectives, they don’t know the data is being served from different data bases.

The main services and purpose are as follows:

  • graphql_federation - This is the services that combines the subgraphs into the supergraph. When a query is sent to the platform, this makes the determination of where the data being requested is located and will make requests to the subgraphs.
  • graphql_data - This service is based on postgraphile and exposes the entities (tables) created by the applications.
  • admin - Backs the admin-ui and exposes administrative tasks, such as creating entities, adding columns, defining business logic, etc.
  • authz - Our authorization service where the policies and decisions are defined for who can access what information/resources.
  • tenant_manager - Provides capabilities for adding new tenants and configuring various settings for the tenants.
  • fhir_graphql_translation - Exposes a FHIR server as a GraphQL graph. This allows FHIR data to be served as part of the super graph.

Repositories

Aside from this repo, there are a few other repositories that make up CommureOS.

  • core-libraries - serves as a shared repository between core-platform and cogwheel (reference implementation). These are common patterns shared by both repositories.
  • deno-scripts - holds CLI tools used in local development. Most importantly, the igor application.
  • Apollo Forks - there are some custom forks of the router that provide additional capabilities for the router – apollo-router – apollo-rs – apollo-rover

The federation server and the forks of apollo are some of the most complicated parts of the architecture. It’s better to understand at a high level their purpose, but to efficiently get started with the platform, it would be better to focus on the other services to start.

Installing the Platform

The main readme has the steps for installing the platform locally. Once the pre-requisites are installed, igor will do most of the work. When running igor start for the first time, it can take some time, and in some cases, timeout. If tilt starts, the best place to check for errors is in the tilt UI for the service in question.

Knowledge Sharing Sessions

There are recorded sessions to help provide context to new developers to the platform. There are a variety of topics covered in the sessions, but by no means comprehensive. If there are topics you feel are missing after going through the session or where more context would be helpful, reach out to the team and we’ll get a session scheduled.

For information specific on the various tools, there is a document for learning resources that might be beneficial for those needing to dive deeper into a technology.

Development Environment

We have a shared development environment deployed to GCP. This is the environment used to support applications developed on CommureOS.

  • Admin UI - Used for managing entities, querying data, adding business logic, etc.
  • /graphql - This is the endpoint all applications will call to make queries to the platform. This endpoint does NOT accept GET requests and does not have UI.

This uses our Okta Preview instance for authentication. This Okta account is not the same as your corporate account and you will need to have an account created. Reach out to someone on the team for help in getting an account created.

Kubeneteres Tools

Everyone in the dev google group should have access to the GCP console (project dev-commureos). If you are unable to access the console, reach out in the team channel and someone will get you set up. Use kubectl to access the cluster from the local CLI. K9s is also a valuable tool that gives a simple UI in the terminal for managing a cluster. If you have access to multiple clusters, kubectx is also helpful.

Getting Help

The platform is complex and additional support might be needed. We’ve started a troubleshooting that contains common errors developers run into building and developing on the platform. If you don’t find a similar issue, try searching slack for the error you are getting. Especially on the first install, it’s likely someone has run into the issue previously. If you’re unable to find a solution to your issue, we have a team channel (#team-platform-api) where people are willing to help work through any errors. It’s important to post in this channel, so that we’re creating documentation on how to solve the issue for future developers.

Prior to the platform being separated from the reference-implementation, many troubleshooting questions were posted to #proj-ref-impl-wg. While new questions should be posted in the #team-platform-api channel, this might be a source to help with troubleshooting.
