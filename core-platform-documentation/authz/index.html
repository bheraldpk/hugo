Authorization

This document describes the CommureOS platform’s authorization model.

Introduction

Authorization is a special kind of business logic that is used to control access to resources and operations. It is often difficult to separate from regular business logic, and there is no universally accepted way to implement it. This document describes how we do it at Commure in the current iteration of our platform.

Authentication vs Authorization

The first thing to know about authorization is how it differs from authentication. Authentication is the process of verifying that a user is who they say they are. Authorization is the process of determining whether or not a user has permission to perform a specific action, such as querying a resource.

In the CommureOS platform, authentication is handled by the Identity Provider (IdP). In most cases this will be Okta, which might delegate to an upstream IdP provided by the customer.

Our platform is designed to support the OIDC protocol, which means that the end result of authentication is most commonly an OpenID Connect id_token in addition to an access_token. The id_token token contains information about the user, including their name, email address, and various other claims that we can configure (e.g. roles or groups).

Components

RFC 2904 defines a set of standard components involved in authorization. These components are shown in the diagram below.

The definitions for each component are as follows.

Policy Enforcement Point (PEP)

The PEP is the component that is responsible for enforcing authorization. Once a decision has been made by the PDP, the PEP is responsible for taking the appropriate action. That may be fetching the resource, or it may returning an error such as of 401 (or 404) to the user.

The individual business services usually acts as the PEP, since the logic for what to do with a given policy decision is usually specific to the service.

Policy Decision Point (PDP)

The PDP is the component that is responsible for making authorization decisions. This involves evaluating the policy logic in the context of the user’s identity and any relevant data available about the user and the action being requested.

This could be implemented by a separate service, as long as it has access to required data to make the decision.

Policy Information Point (PIP)

The PIP is the component that is responsible for providing the data that the PDP needs to make the policy decision. This data could include user data as well as the subset of business data required to make the decision.

Policy Administration Point (PAP)

The PAP is used to administer the policies. Policies are the files that contain the logic for making authorization decisions.

Architecture

Since it can be difficult to separate authorization from regular business logic, authorization has often been implemented as part of the business logic. There are some benefits to this approach. For example, access to the business data required for the decision is easy. Also, the required functionality to take the appropriate action is usually already implemented close by in the code.

The downsides are that the logic to make the decision often has to be implemented multiple times, and changes to the authorization logic can require changes to the code.

Commure Architecture

The Commure authorization architecture attempts to decouple authorization decision-making from individual services as much as possible. This allows for easy changes to the authorization logic, and also allows for the authorization logic to be reused across multiple services. The architecture is shown in the diagram below.

We have decoupled the PIP and PDP into separate services. The PIP is responsible for providing the data that the PDP needs to make the decision. The PDP, which is simply an instance of Open Policy Agent, also acts as the PAP.

The benefits of this approach are that the authorization logic can be changed easily by changing the policies (or in our case, also by changing the data). The logic can also be reused across multiple services. The individual business services don’t need to know how the decision is made, which simplifies their development.

The downsides are that the authorization logic does not have easy access to business data. This means that the PIP (authz) database needs to be populated somehow. The current plan is to use our event-driven architecture for this purpose, but this has not yet been implemented.

Authorization Service

The authorization service is responsible for providing the data to the PDP, and is implemented as a Golang service using all of the libraries and frameworks provided by our scaffolding. The code for this service is in the core-platform repository here.

The two main concepts in the authorization service are rules and data. There are rules for attributes and relationships, and data for attributes and relationships. Only the attribute functionality is currently fully implemented, so for now we will focus on attribute_rules and platform_attributes (attribute data that mirrors existing data in the platform).

Usage

DATA CREATION

All of the functionality of the authorization service is exposed via the REST and gRPC APIs. The easiest way to interact with the service is via the Swagger UI which is available at /q/swagger-ui on the HTTP port. To quickly get an environment to play around with, you can port forward the authz service on GCP dev to your machine:

kubectl -n dev port-forward svc/authz 5051:5051

Then visit http://localhost:5051/q/swagger-u.

The first thing you need to do is create a meta attribute (e.g role). All rules are written against meta attributes. Create one as follows:

// POST /authz/v1/meta-attributes
{
  "id": "f368f91b-bce1-49f6-932f-484e910f1b2a", // IDs will be autogenerated on creation if you leave it empty
  "name": "testRole",
  "description": "Role for testing"
}

To create a rule that says “a user with testRole=admin can read testResource resources”, you would create the following:

// POST /authz/v1/attribute-rules
{
  "attributeId": "f368f91b-bce1-49f6-932f-484e910f1b2a", // references the testRole meta attribute
  "value": "admin", // the value required for the testRole attribute
  "action": "ACTION_READ",
  "resource": {
    "type": "app/testResource",
    "values": []
  },
  "variables": []
}

Now that the meta attribute and rule exist, you can create a platform attribute, which mirrors a piece of data exists somewhere in the platform (or IdP). For example, we can tell the authz service that the https://idp.com/|subject user has the value admin for the f368f91b-bce1-49f6-932f-484e910f1b2a (testRole) meta attribute:

// POST /authz/v1/platform-attributes
{
  "attributeId": "f368f91b-bce1-49f6-932f-484e910f1b2a", // references the the testRole meta attribute
  "userId": "https://idp.com/|subject", // Follows the platform "IdP|sub" format
  "value": "admin",
  "attributeValueSource": "ATTRIBUTE_VALUE_SOURCE_PLATFORM" // more on this in the "sources" section below
}

With the above three resources created, the https://idp.com/|subject user will be able to read app/department resources. The tests contain a lot more examples.

DECISION

Once the authz service has been populated with the data and rules, you can make a request to the PDP (i.e. OPA) to get a decision. Assuming you’ve used the GCP dev environment to create the resources above, we can use the OPA instance running there by first forwarding the port:

kubectl -n dev port-forward svc/opa 8181:80

Then posting to the allow policy endpoint:

// POST https://localhost:8181/v1/data/commure/authz/allow
{
  "input": {
    "user": {
      "token": "eyJhbGciOiAiSFMyNTYiLCAidHlwIjogIkpXVCJ9.eyJhdWQiOiAiYXVkaWVuY2UiLCAiZXhwIjogMTY1NDYzNzg5MiwgImlhdCI6IDE2NTQ2MzQyOTIsICJpc3MiOiAiaHR0cHM6Ly9pZHAuY29tLyIsICJyb2xlcyI6IFsicm9sZUEiLCAicm9sZUIiXSwgInN1YiI6ICJzdWJqZWN0In0.UhaCJHEUSWXrtvjAoC4Vwa3Z4aL9Nro_4FZLPZq80ds"
    },
    "resource": {
      "type": "app/testResource"
    },
    "action": "ACTION_READ"
  }
}

:bulb: You can use the OPA playground to generate an id_token for testing. OPA does not do any id_token validation, so you can use any value you want for any of the claims.

Every decision request must contain a user object with a token field that contains a JWT with at least the iss and sub claims. The input must also have a resource object that must contain at least a type field. The action field must exist and must be one of the ENUM elements defined here.

The above request should give you a response like:

{
  "decision_id": "be464047-253c-4b1b-a109-2ddfd8537282",
  "result": true // <-- this is the decision
}

The actual deployed policies can be found here. There are two policy entry points, one for single decisions (the allow endpoint) and one for bulk decisions (the decisions endpoint). See the tests for more examples.

POLICY LOGIC

Currently, the policy logic is quite simple, and fairly static. It relies on the rules provided by the authz service. See here for the exact policy logic.

The logic is roughly as follows:

  1. Extract the user identity from the input
  2. Get the user attributes from the authz service
  3. Interpolate any values into the attributes
  4. Get the relevant rules from the authz service
  5. Interpolate any values into the rules
  6. Make sure that for every rule, the user has an attribute that satisfies the rule and that the resource constraints are satisfied

CONSTRAINTS

In order to support fine grained authorization, the decision rules can define additional constraints on resources. The resource part of the rule above looked like:

  "resource": {
    "type": "app/testResource",
    "values": []
  },

However, we can add additional constraints to the resource. For example:

    "resource": {
        "type": "app/schedule",
        "values": [
            {"key": "schedule_status_id", "value": "submit"}
        ]
    },

If a rule specifies additional constraints, then the resource submitted with the request must contains all the same keys and values as the rule. Only then will the rule evaluate to true (assuming the other requirements are met). See the tests for plenty of examples.

:bulb: Note that it is up to the business service to send the right values with the decision request. Also, the keys do not have to be top-level keys in the actual resource and the values can be arbitrary JSON objects - OPA will do a deep comparison.

SOURCES

Both rules and attributes can have values interpolated into them at runtime. This is necessary to support all the use cases we need.

Rules

For rules, values are interpolated into the resource field before it is compared to the resource submitted with the decision request. Here is an example rule extracted from the tests:

{
    "id": "aae8d23a-bc23-442a-abea-ba1533aa5894",
    "attributeId": "7b004032-1143-40c3-a47c-d2d0e6005863",
    "value": "approver",
    "action": "ACTION_READ",
    "resource": {
        "type": "test/demo",
        "values": [{
            "key": "constrained_field",
            "value": "constrained_value",
        }],
    },
    "variables": [{
        "field": "field",
        "source": "VARIABLE_SOURCE_ID_TOKEN",
        "idOrPath": "lol",
    }],
},

This rule is granting the approver role the ability to read the test/demo resource type. There is the additional constraint that the constrained_field must be equal to constrained_value.

Also, there is a variable that says that the field field must have the value found in the id_token at path lol. So, if the id_token submitted with the request is:

// id_token
{
  // claims
  "lol": "hehe"
  // other claims
}

The the final resource used when the rule is evaluated would be:

    "resource": {
        "type": "test/demo",
        "values": [{
            "key": "constrained_field",
            "value": "constrained_value",
        },{
            "key": "field",
            "value": "hehe",
        }],
    },

The interpolation code can be found here, and variables can get their values from three different sources:

  • VARIABLE_SOURCE_ATTRIBUTE - the value comes from a user attribute provided by the authz service. In this case the idOrPath field is the UUID of the attribute to use.
  • VARIABLE_SOURCE_ID_TOKEN - the value comes from the id_token. This is the example shown above. In this case the idOrPath field is a JSON path in the id_token. Array indexes are not currently supported.
  • VARIABLE_SOURCE_INPUT - the value of an attribute comes from the OPA input to the decision request. In this case the idOrPath field is a JSON path in the input. Array indexes are not currently supported.

Attributes

Similarly to rules, user attributes can also have values interpolated into them. This is to support use cases where the user attributes are dynamic, and their values are supplied at runtime via the id_token or OPA input. These values are interpolated into the value field of the attribute before the rule is evaluated.

An example from the tests is:

{
    "id": "6f17a26d-4b1b-4519-b333-c1d41a801a79",
    "attributeId": "7b004032-1143-40c3-a47c-d2d0e6005863",
    "userId": "https://commure.idp/authz|sub123nurse",
    "attributeValueSource": "ATTRIBUTE_VALUE_SOURCE_INPUT",
    "path": "inputTestKey",
    // "value": undefined
},

In the above case, the value of input["inputTestKey"] will be interpolated into value field of the attribute, and that value will be used when evaluating the rule.

The interpolation code can be found here, and attributes can get their values from the following sources:

  • ATTRIBUTE_VALUE_SOURCE_PLATFORM - the values comes from the authz service. This is the simplest case, and there is no interpolation. The attribute was created with a value field, and that value is used.
  • ATTRIBUTE_VALUE_SOURCE_INPUT - the value comes from the OPA input to the decision request. In this case the path field is a JSON path in the input. Array indexes are not currently supported.
  • ATTRIBUTE_VALUE_SOURCE_ID_TOKEN - the value comes from the id_token. In this case the path field is a JSON path in the id_token. Array indexes are not currently supported.

TROUBLESHOOTING

Everything has to be exactly correct to have a "result": true response, and it can sometimes be a little tricky to figure out what is wrong. I recommend using the OPA vscode plugin to help with debugging. It allows you to evaluate a policy in the editor, with a specific input provided in an input.json file.

You can then use the builtin print statement to dump the various values. You can even point to the deployed authz service via a port forward by changing the PIP endpoint configuration here.

There is much left to do, both on the authz service and as well as in the business services.

Business Services

graphql-data Fine-Grained Authorization

At the moment, only coarse-grained authorization is implemented. This means that we can only restrict read and write access based on the resource type. We need to extend this to allow for fine-grained authorization, where we can restrict access based on the values of specific fields.

To do this, we will need to extend the function that assembles the authz request to extract field values from the GraphQL query and include them in the decision request.

We also need to implement data filtering in the graphql-data service. We need to figure what use cases we need to support first though.

AuthZ Services

Request Optimization

Currently, there are three request happening for each authorization request, as shown below:

The first request is from the business service to the PDP, and then there are two requests from the PDP to the PIP. One simple optimization would be to combine the second two requests into one, and then have the PIP return the both the relevant rules and attributes together.

Using OPA As A Go Library

Another optimization that we should investigate is using OPA as a Go library. So, instead of having OPA as a separate service, we could evaluate the policies directly inside the AuthZ service. This would combine the PDP and PIP, and eliminate two of the three requests.

The downside is that we would no longer be able to use OPA as a sidecar, and potentially short-circuit decision requests that could be evaluated without any requests to the PIP. We need to clearly define the use cases to determine if this is a good idea or not.
