Authentication

The current GCP dev instance of the platform is configured to use our Okta Preview instance for authentication. This means you will need a separate Okta account (different from your Commure account) to log in. You can ask in the #team-platform-api channel in Slack for an account.

Non-Interactive Login

If you are developing tests, or need to be able to log in without user interaction (but not using a service account), you can use Okta primary authentication.

First, you need to establish an Okta session by sending a request to the /authn endpoint. Start by saving your Okta username and password in environment variables:

export OKTA_USERNAME=your-username
export OKTA_PASSWORD=your-password

Then, send a request to the /authn endpoint and save the sessionToken into a variable called OKTA_SESSION_TOKEN:

export OKTA_SESSION_TOKEN=$(curl -s -X POST \
    'https://commure-preview.oktapreview.com/api/v1/authn' \
    -H 'Accept: application/json' \
    -H 'Content-Type: application/json' \
    -d '{
        "username": "'"$OKTA_USERNAME"'",
        "password": "'"$OKTA_PASSWORD"'"
    }' | jq -r '.sessionToken')

Once you have an established session with Okta, you can go through the normal OAuth 2.0 flow, but you will not be required to enter your credentials into the browser.

The first step is to send a request to the /authorize endpoint to get a code. The expected response is a redirect to the redirect_uri endpoint with the code in the URL as a parameter. However, we will not follow the redirect, but instead just extract the code from the location header in the 302 response and save it in a variable called OKTA_CODE:

export OKTA_CODE=$(curl -s -D - -o /dev/null \
    --request GET "https://commure-preview.oktapreview.com/oauth2/aus18g3ckowuculLe0h8/v1/authorize?client_id=0oa1blmtaggYKFjY70h8&redirect_uri=https://os.dev.commure.com/admin-ui&response_type=code&state=abc123&scope=openid%20profile&sessionToken=$OKTA_SESSION_TOKEN&code_challenge=qjrzSW9gMiUgpUvqgEPE4_-8swvyCtfOVvg55o5S_es&nonce=prettyGoodNonce&code_challenge_method=S256" \
    --header 'Accept: application/json' | grep -i location | sed 's/.*code=\(.*\)&.*/\1/g')

Finally, we can exchange the code for an access_token by sending a request to the /token endpoint:

curl -s --request POST 'https://commure-preview.oktapreview.com/oauth2/aus18g3ckowuculLe0h8/v1/token' \
    --header 'Accept: application/json' \
    --header 'Content-Type: application/x-www-form-urlencoded' \
    --header 'Authorization: Basic MG9hMWJsbXRhZ2dZS0ZqWTcwaDg6cExqcDNjNTBZNDc0VElQX1kxcGd4MDJJaFh6YUtWY2toNEhEY0Jscg==' \
    --data-urlencode "code=$OKTA_CODE" \
    --data-urlencode 'code_verifier=M25iVXpKU3puUjFaYWg3T1NDTDQtcW1ROUY5YXlwalNoc0hhakxifmZHag' \
    --data-urlencode 'grant_type=authorization_code' \
    --data-urlencode 'redirect_uri=https://os.dev.commure.com/admin-ui' | jq

⚠️ WARNING: The Authorization header above contains the applications client secret, so this should not be shared publicly, and the above approach should only be used in the development environment.
